BLOAT

Bloat is for the most part, a very dumb concept nowadays. Most computers of today will not benefit from a few kilobytes of storage/memory saved, no matter how much it stacks up, it's useless. This is not to say we shouldn't be improving software efficiency, that is obviously a good thing to do, my problem starts when we start compromising features for minimal performance improvements, we should be prioritizing features always.

On the opposite side of the spectrum from the bloat-maniacs we have the overwriters, I HATE it when seemingly simple software gets overwritten to the point of ridiculous-ness, I don't always have the time and resources to read through thousands of lines of code to tweak a program that shouldn't have been more than 200 lines long, not to mention a lot of the times it's written in a very complex way that makes it even harder to read and understand, I give GNU coreutils as a prime example here: from a consumer standpoint they're pretty damn great, a bit more heavy than other counterparts performance-wise but very generous with features and in general a great gift from the FSF upon this mortal world, but just *try* looking at the source code to tweak them slightly, it's like reading cross-compiled assembly, not to mention most variable names are wildly unhelpful, here is a easy to visualise example of this [echo.c comparison]. Never dug through the source code but I heard systemd suffers from the same plague, again, from a consumer standpoint you dont get any benefit from lighter equivalents and in the case of systemd all the tweaking you would need you can already do easily, but in most cases of "bloat" just having clearer less useless-ly complex code without sacrificing features would be the golden road to write software IMO.
